# Bruno API Usage for Salesforce

This project includes a Bruno collection under `SalesforceAPI/Salesforce` for quick Salesforce REST API validation outside Playwright.

Use it when you want to:

- validate Salesforce authentication manually
- create sample Accounts or Contacts quickly
- debug API access issues before writing or running Playwright tests
- generate data for UI checks or exploratory testing

---

## Collection Location

```text
SalesforceAPI/
└── Salesforce/
    ├── opencollection.yml
    ├── Token.yml
    ├── Account Creation.yml
    ├── Contact Creation.yml
    ├── Contact Creation and link with Account.yml
    └── environments/
        └── login.yml
```

---

## What Is Included

The Bruno collection currently contains 4 requests:

1. `Token`
   Exchanges Salesforce credentials for an OAuth access token.
2. `Account Creation`
   Creates an `Account` record and stores the generated `account_id`.
3. `Contact Creation`
   Creates a standalone `Contact`.
4. `Contact Creation and link with Account`
   Creates a `Contact` and links it to the previously created `Account`.

The collection also contains one Bruno environment:

- `login`

That environment stores both input secrets and runtime-generated variables.

## Reviewed Behavior

### `Token.yml`

Purpose:

- Calls Salesforce OAuth token endpoint using the password grant flow.
- Saves `access_token` and `instance_url` back into the active Bruno environment.

Important details:

- The login endpoint is hardcoded to `https://login.salesforce.com/services/oauth2/token`
- Auth is driven by these environment variables:
  - `username`
  - `password`
  - `client_id`
  - `client_secret`

Runtime behavior:

- After response, Bruno stores:
  - `access_token`
  - `instance_url`

Assertions already present:

- HTTP status must be `200`
- `access_token` must exist
- `instance_url` must exist
- `token_type` must be `Bearer`
- `issued_at` and `signature` must be present

### `Account Creation.yml`

Purpose:

- Creates a Salesforce `Account`
- Uses the token and instance URL generated by the `Token` request

Runtime behavior:

- Generates a `timestamp` before sending the request
- Sends the account name as `Account using Api {{timestamp}}`
- Saves `account_id` from the response

Assertions already present:

- HTTP status must be `200` or `201`
- response must contain an `id`
- `id` must start with `001`
- `success` must be `true`
- `errors` must be an empty array

### `Contact Creation.yml`

Purpose:

- Creates an independent Salesforce `Contact`

Runtime behavior:

- Generates a `timestamp`
- Sends:
  - `FirstName: Created from {{timestamp}}`
  - `LastName: Bruno {{timestamp}}`

Assertions already present:

- HTTP status must be `200` or `201`
- response must contain an `id`
- `id` must start with `003`
- `success` must be `true`
- `errors` must be an empty array

### `Contact Creation and link with Account.yml`

Purpose:

- Creates a `Contact` linked to an existing `Account`

Runtime behavior:

- Generates a `timestamp`
- Uses `{{account_id}}` from the environment
- Sends `AccountId` in the request body

Assertions already present:

- HTTP status must be `200` or `201`
- response must contain an `id`
- `id` must start with `003`
- `success` must be `true`
- `errors` must be an empty array

Important dependency:

- This request expects `account_id` to already exist in the selected Bruno environment.
- In the current setup, that value is typically produced by running `Account Creation` first.

## Environment Variables

The Bruno environment file is `SalesforceAPI/Salesforce/environments/login.yml`.

It defines these variables:

| Variable | Type | Purpose |
| --- | --- | --- |
| `username` | secret | Salesforce username |
| `password` | secret | Salesforce password or password+token, depending on org setup |
| `client_id` | secret | Connected App consumer key |
| `client_secret` | secret | Connected App consumer secret |
| `access_token` | secret | Generated by `Token` |
| `instance_url` | secret | Generated by `Token` |
| `account_id` | secret | Generated by `Account Creation` |

There is also one runtime-only variable created inside request scripts:

| Variable | Source | Purpose |
| --- | --- | --- |
| `timestamp` | request script | Makes record names unique |

## Recommended Usage Flow

### 1. Open the Collection in Bruno

Open Bruno and import or open the collection root:

```text
SalesforceAPI/Salesforce
```

Bruno should detect the collection from `opencollection.yml`.

### 2. Select the `login` Environment

Choose the `login` environment before running any request.

Populate these values:

- `username`
- `password`
- `client_id`
- `client_secret`

Do not manually set these unless you are intentionally overriding runtime behavior:

- `access_token`
- `instance_url`
- `account_id`

### 3. Run `Token`

Run `Token` first.

Expected result:

- successful `200` response
- `access_token` saved into environment
- `instance_url` saved into environment

If this step fails, the other requests will also fail.

### 4. Run a Data-Creation Request

Choose based on your need:

- Run `Contact Creation` to create only a contact
- Run `Account Creation` to create an account
- Run `Contact Creation and link with Account` only after `Account Creation`

### 5. For Linked Contact Flow, Keep the Order

Use this order:

1. `Token`
2. `Account Creation`
3. `Contact Creation and link with Account`

That sequence ensures `account_id` is available for the linked-contact request.

## How the Request Chaining Works

The collection relies on Bruno runtime scripts and environment variables.

Flow:

1. `Token` stores `access_token` and `instance_url`
2. `Account Creation` uses those values and stores `account_id`
3. `Contact Creation and link with Account` uses all three generated values

This means the collection is stateful within the selected environment.

If you clear environment values or switch environments, rerun the dependency requests.

Run only:

Run:

Run:

1. `Token`

3. `Contact Creation and link with Account`

## Relationship to Playwright Automation

The Bruno collection is useful for manual API validation, but it is separate from the main automated framework.

Current split:

- `SalesforceAPI/Salesforce` is for Bruno-based manual or exploratory API usage
- `tests/api` is for Playwright-based automated API tests
- `tests/e2e` is for hybrid flows where API-created data is validated through UI

Bruno is especially helpful when:

- debugging request payloads quickly
- validating credentials outside the test framework
- reproducing an API issue without running the full Playwright suite

## Current Limitations Observed During Review

These are not blockers, but they are worth knowing when you use the collection.

### 1. Login URL Is Hardcoded

`Token.yml` always calls:

```text
https://login.salesforce.com/services/oauth2/token
```

If your team needs sandbox-specific login behavior such as `test.salesforce.com`, the request will need to be updated.

### 2. API Version Is Hardcoded

The record-creation requests use:

```text
/services/data/v66.0/
```

If your Salesforce org standards change, update all Bruno requests consistently.

### 3. Linked Contact Depends on Prior State

`Contact Creation and link with Account` requires `account_id` to already be populated in the environment.

If you skip `Account Creation`, that request will fail.

### 4. Contact ID Is Not Persisted

The contact requests validate the returned `id`, but they do not currently save a `contact_id` back into the environment.

That is fine for the current collection, but it limits downstream request chaining.

## Troubleshooting

### `Token` fails with auth errors

Check:

- `username` is correct
- `password` format matches your org requirements
- `client_id` and `client_secret` come from the correct Connected App
- the Connected App allows the OAuth password flow required by this request

### Create requests fail with unauthorized errors

Usually means:

- `Token` was not run first
- `access_token` expired
- `instance_url` is missing or stale

Run `Token` again and retry.

### Linked contact request fails

Check:

- `Account Creation` was run successfully first
- `account_id` exists in the active environment
- the active environment is still `login`

### Salesforce returns validation or permission errors

Check:

- field-level or object-level permissions for the API user
- org validation rules
- required fields in your Salesforce setup

## Security Guidance

- Treat Bruno environment values as sensitive because they contain Salesforce credentials and tokens
- Do not commit populated secret values into version control
- Prefer using Bruno secrets/environment handling rather than hardcoding credentials in requests
- Regenerate tokens by rerunning `Token` instead of copying tokens manually between tools
